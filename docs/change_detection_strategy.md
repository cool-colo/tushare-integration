# API 限速场景下的数据变更发现方案

## 1. 背景

当前一期数据源以 Tushare 为主，存在两个现实约束：

- API 有频率限制
- 日终窗口有限，模型训练和第二天推理都要抢时间

因此，**不能采用“每天全量拉所有表”** 的策略。  
但另一方面，很多表又可能发生历史修订、补发、状态变化，如果完全不做历史回补，就会出现：

- 训练样本和实际可见信息不一致
- 回测存在未来函数风险
- 第二天策略输入口径漂移

所以这里要解决的问题不是“如何全量拉”，而是：

> **在不做全量的前提下，如何尽可能发现上游数据变更，并把变化版本保留下来。**

---

## 2. 核心判断

需要先讲清楚一个事实：

> 如果上游没有提供 `updated_at`、版本号、CDC 日志或按更新时间过滤的能力，就不可能在“不重拉数据”的前提下 100% 精确知道哪些历史记录被修改了。

Tushare 的大多数接口并不提供标准 CDC。  
因此我们不能设计成：

- 先知道哪里变了
- 再精准拉那几行

而应该设计成：

- 根据表的业务特征，缩小“最可能变化的范围”
- 对这个范围做策略性重拉
- 在本地识别“这次和上次是否不同”

这就是本方案的设计基础。

---

## 3. 总体方案

建议采用四层策略：

1. 表分型
2. 热窗口回补
3. 本地哈希比对
4. 低频兜底全扫

整体思路如下：

```text
判断表类型
  -> 决定增量方式
  -> 设定回补窗口
  -> 重拉候选范围
  -> 本地做主键+哈希比对
  -> 有变化则生成新版本
  -> 无变化则跳过
```

---

## 4. 表分型策略

不要对所有表使用同一种增量规则。  
建议至少分为四类。

## 4.1 A 类：交易日快照型

典型表：

- 股票/指数/期货日线
- 日频资金流
- 龙虎榜
- 大宗交易
- 涨停专题

特征：

- 以 `trade_date` 为核心业务时间
- 一天一份快照
- 历史发生修订的概率相对低，但不是零

推荐策略：

- 只拉新的 `trade_date`
- 再回补最近 `3~10` 个交易日
- 对相同主键做哈希比对

适合表举例：

- `daily`
- `index_daily`
- `fut_daily`
- `daily_basic`
- `moneyflow`
- `top_list`
- `top_inst`
- `block_trade`
- `limit_list_d`

## 4.2 B 类：公告 / 财务 / 事件型

典型表：

- 三大财报
- 财务指标
- 业绩预告/快报
- 分红
- 回购
- 质押
- 股东人数
- 股东增减持

特征：

- 以 `ann_date`、`end_date`、`begin_date` 等业务时间为主
- 存在补发、修订、状态更新
- 只拉“今天新增”很容易漏历史变化

推荐策略：

- 每天按公告窗口回扫最近 `90~365` 天
- 月度或季度再做一次更长窗口兜底
- 用主键 + 哈希识别变化

适合表举例：

- `balancesheet`
- `cashflow`
- `income`
- `fina_indicator`
- `forecast`
- `express`
- `dividend`
- `repurchase`
- `pledge_detail`
- `stk_holdernumber`
- `stk_holdertrade`

## 4.3 C 类：主数据 / 名单 / 成员关系型

典型表：

- 股票列表
- 公司信息
- 指数基础信息
- 成分关系
- 港股通/沪深股通名单

特征：

- 更新频率不高
- 但一旦变化，对 universe、风控、回测口径影响很大

推荐策略：

- 日常只扫描近期可能变动的范围
- 周度或月度做一次全量刷新
- 如果存在 `in_date/out_date/list_date/delist_date`，优先围绕这些边界扫

适合表举例：

- `stock_basic`
- `stock_company`
- `index_basic`
- `hs_const`
- `index_member`
- `ths_member`
- `concept_detail`

## 4.4 D 类：小表 / 字典表

典型表：

- 交易日历
- 概念分类
- 行业分类
- 名录表

特征：

- 表小
- 计算和存储成本低

推荐策略：

- 直接全量刷新

适合表举例：

- `trade_cal`
- `concept`
- `index_classify`
- `hm_list`

---

## 5. 热窗口回补

热窗口回补是当前阶段最重要的能力。

原因：

- 你事先不知道上游改了哪一天
- 但绝大多数变化都集中在“最近一段时间”

建议每张表都定义两个窗口：

- `new_window`
- `repair_window`

示例：

| 表类型 | new_window | repair_window |
|---|---|---|
| 交易日快照型 | 当日新交易日 | 最近 5 个交易日 |
| 资金流/榜单类 | 当日新交易日 | 最近 5~10 个交易日 |
| 财务/公告/事件类 | 当日新公告窗口 | 最近 90~180 天 |
| 主数据/名单类 | 当日疑似变化范围 | 最近 30 天 + 周度全量 |
| 小表/字典类 | 无需单独定义 | 直接全量 |

这一步的本质是：

> 用少量额外请求，换取对高概率变更的覆盖能力。

---

## 6. 本地哈希比对

既然上游不给可靠变更标记，就必须在本地做。

建议给 ODS 层增加以下元数据字段：

- `_batch_id`
- `_ingest_time`
- `_record_hash`

其中 `_record_hash` 的计算方式建议是：

- 对业务字段按固定顺序序列化
- 计算 `md5` 或 `sha1`

这样当某个业务主键被重新拉取时，可以判断：

- 主键相同，哈希相同：无变化
- 主键相同，哈希不同：有变化
- 主键不存在：新增

这是当前阶段最简单、最有效的“变更发现”手段。

---

## 7. 低频兜底全扫

热窗口不是万能的。

有些接口可能：

- 很晚才修历史
- 源侧补数节奏不稳定
- 某些季度或年报数据会被长期回溯修正

所以还需要低频兜底任务。

建议频率：

- 周度全扫：主数据、名单、成员关系
- 月度全扫：事件、公告重点表
- 季度全扫：三大报表、财务指标
- 小表：直接日度全量

这些任务不应占用日终主链路，可安排在：

- 深夜
- 周末
- 非交易时段

---

## 8. 推荐的三条任务链路

## 8.1 日终主链路

目标：

- 保障模型训练和第二天推理时效

执行内容：

1. 先跑交易日快照型表
2. 再跑当天决策必须用到的公告/事件热窗口
3. 生成 DWD 和服务快照
4. 启动模型训练和策略生成

## 8.2 日终修复链路

目标：

- 不阻塞主链路的前提下修复最近历史变化

执行内容：

1. 回补最近 `N` 日热窗口
2. 做哈希比对
3. 如果发现变化，则生成新版本并驱动 DWD 更新

## 8.3 低频兜底链路

目标：

- 捕获热窗口之外的迟到修订

执行内容：

- 周/月/季频错峰全扫

推荐的整体时序：

```text
收盘后
  -> 日终主链路
  -> 产出训练/推理所需快照
  -> 日终修复链路补最近窗口
  -> 深夜/周末运行低频兜底全扫
```

## 9. 推荐的表级配置方式

建议把这套策略参数化，而不是散落在 spider 代码里。

示例：

```yaml
table: daily
change_detection:
  mode: trade_date_incremental
  repair_window_days: 5
  full_refresh_cron: "0 3 * * 0"
```

```yaml
table: fina_indicator
change_detection:
  mode: ann_date_rolling
  repair_window_days: 180
  full_refresh_cron: "0 4 1 * *"
```

```yaml
table: stock_basic
change_detection:
  mode: low_freq_full_refresh
  repair_window_days: 30
  full_refresh_cron: "0 2 * * 0"
```

建议至少配置：

- `mode`
- `repair_window_days`
- `full_refresh_cron`
- `priority`
- `blocking_for_t_plus_1`

其中：

- `priority` 用于调度排序
- `blocking_for_t_plus_1` 用于决定是否阻塞当晚训练/推理

## 10. 与 PIT 的关系

这部分很关键。

> 做 PIT 不等于每天全量拉所有表。  
> 真正决定 PIT 是否成立的，是“变化版本有没有被识别并保留下来”。

只要做到以下几点，就可以在“非全量拉取”的条件下构建 PIT：

- 对可能变更的范围进行策略性重拉
- 用哈希识别记录变化
- 不覆盖旧版本，而是保留新版本
- 在 DWD 层按 `available_trade_date + version` 组织数据

所以：

- **不全量拉** 不是问题
- **只保留 latest、看不到历史版本** 才是问题

---

## 11. 当前阶段的推荐方案

如果按“简单、高效、可尽快落地”来选，建议直接采用下面这套：

- 小表直接全量
- 交易日快照表按新日期增量 + 最近 `5` 天回补
- 财务/公告/事件表按 `ann_date` 热窗口回扫 + 周/月兜底
- 所有重拉结果做 `_record_hash` 比对
- 发现变化时生成新版本，不覆盖旧版本
- 日终主链路只跑“第二天必须用到”的表
- 其余历史修复放进修复链路和兜底链路

这套方案兼顾了：

- API 频控约束
- 日终时效要求
- 变更发现能力
- PIT 建设要求

---

## 12. 最终结论

在 Tushare 这类没有标准 CDC 的源上，最合理的工程策略不是“全量拉”或者“完全精准增量”，而是：

> **分表型增量 + 热窗口回补 + 本地哈希比对 + 低频兜底全扫**

这是当前阶段最简单、最高效、也最现实可落地的方案。
