# 高维因子复权存储方案（增强版）

## 1. 背景与目标

在量化投资、因子研究、策略回测与模型训练场景中，价格序列必须具备严格的前后可比性。股票在生命周期内会发生分红、送股、转增、配股、拆股、合并等除权除息事件，如果直接使用原始行情字段（`open`、`high`、`low`、`close`、`volume`、`amount`），价格序列会出现非市场交易行为导致的断裂。

这类断裂会直接影响：

1. 收益率计算的准确性；
2. 技术指标的连续性；
3. 价量类因子的稳定性；
4. 回测结果的可复现性；
5. 训练样本与线上推理特征的一致性；
6. 不同研究员、不同任务之间的数据口径一致性。

因此，复权不是简单的数据展示需求，而是量化研究基础数据层必须解决的核心问题。

本文目标是设计一套适用于“高维因子 + 大规模股票池 + 多年历史数据 + 多任务并发”的复权因子存储方案，在保证 Point-in-Time（PIT，时点一致性）和结果可复现的前提下，尽可能降低计算成本、存储成本和数据维护复杂度。

---

## 2. 为什么必须复权

股票除权除息会导致价格在事件日出现机械性跳变。例如某股票每 10 股送 10 股，理论上股价会接近折半。如果不做复权处理，价格序列会表现为一次巨大的下跌，但这并不代表真实的市场亏损。

若直接基于未复权价格计算因子，会产生以下问题：

1. **收益率失真**  
   除权日价格变化被误识别为真实收益，导致日收益率、累计收益率、波动率等指标异常。

2. **技术指标失效**  
   均线、MACD、RSI、布林带、突破类指标会受到价格断点影响，产生虚假信号。

3. **机器学习样本污染**  
   模型会学习到由企业行为造成的伪价格波动，而非真实市场行为。

4. **回测不可比较**  
   不同时间点拉取的数据可能存在不同的复权口径，导致同一策略在不同时间运行结果不一致。

5. **线上线下不一致**  
   如果训练、回测、推理阶段使用的复权方式不一致，会导致特征分布漂移，影响模型稳定性。

因此，对于量化因子系统，复权处理应当作为行情数据标准化层的一部分，而不是临时计算逻辑。

---

## 3. 为什么不适合完全实时计算

高维因子系统通常具有以下特点：

- 股票数量：数千只 A 股或全市场证券；
- 时间跨度：多年甚至十年以上日频、分钟级或更高频历史数据；
- 因子数量：几十到几百个基础因子，甚至上千个衍生因子；
- 表达式复杂度：包含滚动窗口、截面标准化、行业中性化、滞后项、条件过滤等逻辑；
- 使用场景：训练、回测、参数搜索、线上推理、多研究员并发访问。

若每次任务都实时执行以下流程：

```text
原始行情 -> 复权价格 -> 因子表达式计算 -> 截面处理 -> 模型/回测使用
```

会带来明显问题：

1. **计算成本高**  
   几千只股票 × 多年历史 × 几百个因子 × 多个窗口，实时计算需要消耗大量 CPU、内存和 IO。

2. **任务延迟高**  
   回测和训练任务会被数据准备阶段显著拖慢，影响研究效率。

3. **并发能力差**  
   多个研究员或多个批量任务同时运行时，容易重复计算同一批历史因子。

4. **结果难以复现**  
   如果复权因子或基础行情被修订，实时计算结果可能随时间变化。

5. **数据治理困难**  
   缺乏清晰的版本边界，不利于排查历史结果差异。

因此，高维因子系统应采用“预计算 + 版本化 + 增量更新 + PIT 查询”的存储架构，而不是完全依赖实时计算。

## 4. 复权方式选择

常见复权方式包括后复权、前复权和不复权。不同方式适用于不同场景。

### 4.1 后复权

后复权通常以历史价格为基准，将后续价格向上调整，使历史价格保持原始水平，当前价格可能高于真实交易价格。

常见计算形式：

```text
后复权价格 = 原始价格 × 复权因子
```

#### 优点

1. 计算逻辑简单；
2. 适合长期收益分析；
3. 便于观察长期持有收益；
4. 通常只需维护一份最新全量结果；
5. 对历史累计收益展示较直观。

#### 缺点

1. 当前价格不等于真实可交易价格；
2. 历史价格可能被放大到异常高的水平；
3. 不适合短期技术分析和交易信号生成；
4. 与实际交易成本、涨跌停价格、盘口价格不一致；
5. 对交易执行和短周期回测的解释性较弱。

#### 适用场景

- 长期收益表现展示；
- 净值曲线分析；
- 长周期持有收益比较；
- 部分平台的因子展示口径。

例如，聚宽部分因子接口采用类似后复权或基于复权价格的处理方式：  
<https://www.joinquant.com/help/api/help#name:factor_values>

### 4.2 前复权

前复权通常以当前价格为基准，将历史价格向下调整，使最新价格等于真实交易价格。

#### 优点

1. 最新价格与市场真实交易价格一致；
2. 更符合交易决策和短期技术分析习惯；
3. 技术指标形态较连续；
4. 适合策略回测、择时信号、价量因子计算。

#### 缺点

1. 历史前复权价格会随新的除权除息事件变化；
2. 如果直接使用最新前复权全量数据回测，会引入未来信息；
3. 不同日期生成的同一历史日期价格可能不同；
4. 回测、训练和实盘推理之间容易出现数据口径不一致。

#### 适用场景

- 技术分析；
- 短中期交易策略；
- 价量因子计算；
- 机器学习特征工程；
- 需要最新价格保持真实交易含义的场景。

### 4.3 不复权

不复权价格保持交易所原始价格，不进行连续性调整。

#### 优点

1. 与真实交易价格完全一致；
2. 适合交易执行、成交价校验、盘口分析；
3. 不存在复权口径争议。

#### 缺点

1. 历史价格序列不连续；
2. 不适合直接计算长期收益率和技术指标；
3. 除权除息事件会污染因子信号。

#### 适用场景

- 订单成交分析；
- 实盘撮合；
- 涨跌停判断；
- 交易成本计算；
- 原始行情归档。

## 5. 前复权存储方案对比

由于前复权更适合高维价量因子和交易策略研究，因此本文重点讨论前复权数据的存储方案。

### 5.1 方案一：实时计算

每次查询或任务执行时，基于原始行情和当时复权因子实时生成前复权价格，再计算因子。

#### 优点

1. 存储成本低；
2. 理论上可按任意时点动态生成；
3. 不需要维护大量派生数据表。

#### 缺点

1. 计算成本极高；
2. 不适合反复回测和参数搜索；
3. 不适合多研究员并发；
4. 查询延迟不可控；
5. 对计算引擎和缓存系统依赖重；
6. 一旦底层复权因子修订，历史任务复现困难。

#### 结论

该方案适合小规模研究或临时验证，不适合作为高维因子生产系统的主方案。

### 5.2 方案二：永远使用最新前复权全量数据

系统每天根据最新复权因子重算全部历史前复权价格和相关因子，只保留一份最新结果。

#### 优点

1. 实现简单；
2. 查询性能好；
3. 存储量相对可控；
4. 数据展示口径统一。

#### 缺点

1. 历史数据不可复现；
2. 回测存在未来函数风险；
3. 训练样本可能使用了未来才知道的复权信息；
4. 策略历史表现会随着时间推移发生变化；
5. 无法解释旧回测结果与新回测结果之间的差异。

#### 结论

该方案适合数据展示和非严格研究场景，不适合严肃量化回测、模型训练和实盘前验证。

### 5.3 方案三：保存每日完整快照

每天保存截至当日可见的全市场前复权价格和所有因子结果。回测时按回测日期读取对应快照。

#### 优点

1. 严格满足 Point-in-Time；
2. 回测结果可复现；
3. 数据口径清晰；
4. 不易引入未来函数。

#### 缺点

1. 存储成本极高；
2. 每日全量快照会产生大量重复数据；
3. Backfill 时间长；
4. 数据修复复杂；
5. 因子数量越多，存储膨胀越严重；
6. 对分区、压缩、冷热分层要求高。

#### 结论

该方案最严谨，但成本过高。除非监管、审计或超高复现要求非常强，否则不建议对所有高维因子保存完整每日快照。

### 5.4 方案四：PIT 版本化增量存储（推荐）

推荐采用“原始行情不变 + 复权因子版本化 + 派生因子增量物化 + 查询时按 as-of 口径还原”的混合方案。

核心思想是：

```text
不保存每日全量快照，而是保存足以还原任意历史时点可见数据的版本信息和增量变更。
```

该方案将数据分为四层：

1. 原始行情层；
2. 复权因子 PIT 层；
3. 前复权价格物化层；
4. 高维因子结果层。

通过 `trade_date` 与 `asof_date` 双时间维度，实现可复现查询。

## 6. 推荐架构设计

## 6.1 数据分层

### 6.1.1 原始行情层：Raw Market Data

用于保存交易所或供应商提供的原始行情数据。该层原则上只追加、不覆盖。

建议字段：

| 字段 | 含义 |
|---|---|
| symbol | 证券代码 |
| trade_date | 交易日期 |
| open | 原始开盘价 |
| high | 原始最高价 |
| low | 原始最低价 |
| close | 原始收盘价 |
| volume | 成交量 |
| amount | 成交额 |
| source | 数据来源 |
| ingest_time | 入库时间 |
| data_version | 数据版本 |

关键要求：

1. 原始行情必须可追溯；
2. 数据供应商修订时不应直接覆盖旧值，应保留版本；
3. 所有派生数据必须能够回溯到原始行情版本。

### 6.1.2 复权因子层：Adjustment Factor PIT Store

用于保存复权因子的时点版本。

建议字段：

| 字段 | 含义 |
|---|---|
| symbol | 证券代码 |
| trade_date | 被复权价格对应的交易日 |
| adj_factor | 复权因子 |
| asof_date | 该复权因子在系统中可见的日期 |
| event_date | 除权除息事件日期 |
| event_type | 事件类型，如分红、送股、配股 |
| source | 数据来源 |
| version | 版本号 |
| is_latest | 是否为当前最新版本 |
| created_at | 创建时间 |

其中：

- `trade_date` 表示数据作用于哪个交易日；
- `asof_date` 表示在什么日期之后，系统或研究员可以看到该版本；
- 查询历史回测数据时，必须使用 `asof_date <= 回测当日` 的最新版本。

示例查询语义：

```sql
SELECT *
FROM adj_factor_pit
WHERE symbol = '000001.SZ'
  AND trade_date <= :trade_date
  AND asof_date <= :asof_date
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY symbol, trade_date
  ORDER BY asof_date DESC, version DESC
) = 1;
```

### 6.1.3 前复权价格物化层：Adjusted Price Store

用于保存常用前复权价格，降低因子计算成本。

建议字段：

| 字段 | 含义 |
|---|---|
| symbol | 证券代码 |
| trade_date | 交易日期 |
| asof_date | 可见日期 |
| adj_open | 前复权开盘价 |
| adj_high | 前复权最高价 |
| adj_low | 前复权最低价 |
| adj_close | 前复权收盘价 |
| adj_volume | 复权成交量，可选 |
| price_version | 价格版本 |
| adj_factor_version | 使用的复权因子版本 |
| created_at | 创建时间 |

是否全量保存 `adj_price`，取决于成本和查询性能要求。对于日频数据，建议保存；对于分钟级或 Tick 数据，应优先考虑按需生成或只保存关键窗口数据。

### 6.1.4 高维因子结果层：Factor Feature Store

用于保存预计算后的因子值。

建议字段：

| 字段 | 含义 |
|---|---|
| symbol | 证券代码 |
| trade_date | 因子对应交易日 |
| asof_date | 因子在系统中可见的日期 |
| factor_name | 因子名称 |
| factor_value | 因子值 |
| factor_version | 因子表达式版本 |
| input_price_version | 输入价格版本 |
| input_adj_factor_version | 输入复权因子版本 |
| calc_time | 计算时间 |
| data_quality_flag | 数据质量标记 |

对于高维因子，应优先采用列式存储格式，例如 Parquet、ORC 或 ClickHouse MergeTree 表，以提高压缩率和批量扫描性能。

## 7. 前复权计算口径

前复权的目标是保持最新价格等于真实交易价格，同时调整历史价格，使价格序列连续。

设：

- `P_raw(t)`：交易日 `t` 的原始价格；
- `AF(t, A)`：在 `asof_date = A` 时可见的交易日 `t` 的复权因子；
- `T`：截至 `A` 时的最新交易日；
- `AF(T, A)`：最新交易日复权因子。

则前复权价格可表示为：

```text
P_forward_adj(t, A) = P_raw(t) × AF(t, A) / AF(T, A)
```

其中 `A` 是关键变量。对于任意历史回测日期，只能使用当时已经可见的复权因子，而不能使用当前最新复权因子。

若忽略 `A`，直接用今天的最新复权因子生成历史价格，就会引入未来函数。

## 8. PIT 查询原则

所有训练、回测和研究查询必须明确两个时间：

1. `trade_date`：数据对应的市场交易日；
2. `asof_date`：研究者或系统在该日期可见的数据版本。

查询规则：

```text
对于任意 trade_date = D 的数据，
在 asof_date = A 的回测中，
只能使用 A 当日及之前已经发布、入库、可见的数据版本。
```

即：

```text
visible_version = max(version where asof_date <= A)
```

对于回测任务，通常设置：

```text
asof_date = 回测当前日期
```

对于模型训练任务，可设置：

```text
asof_date = 训练样本生成日期
```

对于线上推理任务，可设置：

```text
asof_date = 当前生产日期
```

## 9. 存储策略

### 9.1 不建议存储所有每日全量快照

每日完整快照虽然最容易理解，但会造成严重的数据膨胀。尤其在以下条件下，存储成本不可接受：

```text
股票数量 × 交易日数量 × 因子数量 × 快照日期数量
```

例如：

```text
5,000 只股票 × 3,000 个交易日 × 500 个因子 × 3,000 个快照日期
```

该规模会产生万亿级别数据点，大量内容只是重复快照。

### 9.2 推荐存储“版本差异”而不是“每日全量”

复权因子的变化主要发生在除权除息事件之后，并非每天所有股票都变化。因此可以只保存变化事件和版本差异。

推荐方式：

1. 原始行情全量保存；
2. 复权因子按 `symbol + trade_date + asof_date + version` 保存；
3. 前复权价格只对受影响股票和受影响日期区间增量重算；
4. 因子结果只对受影响窗口增量重算；
5. 查询时通过 `asof_date` 选择正确版本。

### 9.3 分区建议

推荐分区方式：

| 数据层 | 推荐分区 |
|---|---|
| 原始行情 | `trade_date`，可辅助按 `symbol_hash` 分桶 |
| 复权因子 | `asof_date` 或 `event_date` |
| 前复权价格 | `asof_date` + `trade_date`，或 `trade_date` + `version` |
| 因子结果 | `factor_name` + `trade_date`，或 `trade_date` + `factor_group` |

对于列式存储，应避免单文件过小，建议按交易日、因子组、股票分桶控制文件大小。

### 9.4 冷热分层

建议将数据分为：

1. **热数据**：最近 1-2 年，供频繁回测、训练和推理使用；
2. **温数据**：2-5 年历史，供研究和长周期回测使用；
3. **冷数据**：更早历史，仅用于归档和低频查询。

热数据可放在 ClickHouse、DuckDB、Iceberg 热分区或高性能对象存储；冷数据可使用 Parquet + 对象存储归档。

## 10. 增量更新流程

每日数据更新建议采用以下流程：

```text
1. 拉取原始行情
2. 拉取 corporate action / 除权除息数据
3. 生成或更新复权因子 PIT 版本
4. 识别受影响股票和日期区间
5. 增量重算前复权价格
6. 识别受影响因子窗口
7. 增量重算因子值
8. 写入质量校验结果
9. 发布数据版本
10. 记录数据血缘和审计日志
```

### 10.1 受影响区间识别

一次除权除息事件通常会影响该股票事件日前的历史前复权价格。因此不应全市场重算，而应定位：

```text
affected_symbols = 发生复权因子变化的股票
affected_trade_dates = 该股票历史价格中因复权因子变化而需要调整的日期范围
affected_factors = 依赖价格字段的因子集合
affected_windows = 因子滚动窗口覆盖的日期范围
```

例如，一个 60 日均线因子在某日价格被修订时，至少需要重算该日之后 60 个交易日内的因子值。

### 10.2 数据发布机制

建议采用两阶段发布：

1. **Staging 层**：写入新版本数据并完成校验；
2. **Production 层**：校验通过后，将版本标记为可见。

这样可以避免用户读取到半更新状态的数据。

---

## 11. 数据质量校验

必须对复权数据和因子结果进行系统性校验。

### 11.1 行情校验

1. `high >= max(open, close, low)`；
2. `low <= min(open, close, high)`；
3. `volume >= 0`；
4. `amount >= 0`；
5. 停牌日价格处理规则一致；
6. 涨跌停边界与原始价格一致。

### 11.2 复权因子校验

1. 复权因子必须为正数；
2. 同一 `symbol + trade_date + asof_date` 不应存在多个可见版本；
3. 除权除息事件日前后因子变化应与事件类型匹配；
4. 新旧版本差异超过阈值时触发告警；
5. 复权后价格序列不应出现非事件驱动的异常跳变。

### 11.3 因子结果校验

1. 空值比例监控；
2. 极值比例监控；
3. 横截面分布漂移监控；
4. 与上一版本相关性对比；
5. 关键因子的回测 sanity check；
6. 因子 IC、Rank IC、覆盖率稳定性监控。

---

## 12. 版本管理与血缘追踪

为了保证可复现性，每次因子计算都必须记录输入版本。

推荐记录：

```text
factor_run_id
factor_name
factor_version
code_commit_id
raw_data_version
adj_factor_version
price_version
calendar_version
universe_version
industry_classification_version
asof_date
calc_time
operator / job_id
```

其中 `factor_version` 不仅应包含因子名称，还应包含表达式、参数、窗口长度、缺失值处理方式、中性化方式、标准化方式等信息。

如果任一输入版本发生变化，都应生成新的因子版本或新的计算批次，而不应静默覆盖历史结果。

## 13. 推荐表结构示例

### 13.1 复权因子表

```sql
CREATE TABLE adj_factor_pit (
    symbol                  String,
    trade_date              Date,
    asof_date               Date,
    adj_factor              Float64,
    event_date              Nullable(Date),
    event_type              Nullable(String),
    source                  String,
    version                 UInt64,
    is_latest               UInt8,
    created_at              DateTime
)
ENGINE = MergeTree
PARTITION BY toYYYYMM(asof_date)
ORDER BY (symbol, trade_date, asof_date, version);
```

### 13.2 前复权价格表

```sql
CREATE TABLE adjusted_price_pit (
    symbol                  String,
    trade_date              Date,
    asof_date               Date,
    adj_open                Float64,
    adj_high                Float64,
    adj_low                 Float64,
    adj_close               Float64,
    adj_volume              Nullable(Float64),
    raw_data_version        UInt64,
    adj_factor_version      UInt64,
    price_version           UInt64,
    created_at              DateTime
)
ENGINE = MergeTree
PARTITION BY toYYYYMM(trade_date)
ORDER BY (symbol, trade_date, asof_date, price_version);
```

### 13.3 因子结果表

```sql
CREATE TABLE factor_feature_pit (
    symbol                  String,
    trade_date              Date,
    asof_date               Date,
    factor_name             LowCardinality(String),
    factor_value            Float64,
    factor_version          String,
    input_price_version     UInt64,
    input_adj_factor_version UInt64,
    calc_run_id             String,
    data_quality_flag       UInt8,
    created_at              DateTime
)
ENGINE = MergeTree
PARTITION BY (toYYYYMM(trade_date), factor_name)
ORDER BY (factor_name, trade_date, symbol, asof_date);
```

实际生产中也可以采用宽表或数组列结构，例如：

```text
symbol, trade_date, asof_date, factor_vector, factor_version_set
```

宽表查询性能更好，但版本治理和增量更新更复杂；长表灵活性更好，但查询和存储开销更高。建议根据因子数量、查询模式和存储引擎做权衡。

## 14. 查询示例

### 14.1 查询某回测日可见的前复权价格

```sql
SELECT *
FROM adjusted_price_pit
WHERE trade_date BETWEEN :start_date AND :end_date
  AND asof_date <= :backtest_date
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY symbol, trade_date
    ORDER BY asof_date DESC, price_version DESC
) = 1;
```

### 14.2 查询某回测日可见的因子值

```sql
SELECT *
FROM factor_feature_pit
WHERE factor_name IN (:factor_names)
  AND trade_date = :trade_date
  AND asof_date <= :backtest_date
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY symbol, trade_date, factor_name
    ORDER BY asof_date DESC, created_at DESC
) = 1;
```

### 14.3 查询最新生产版本因子

```sql
SELECT *
FROM factor_feature_pit
WHERE factor_name IN (:factor_names)
  AND trade_date = :trade_date
  AND asof_date = (
      SELECT max(asof_date)
      FROM factor_feature_pit
      WHERE trade_date = :trade_date
  );
```

## 15. 推荐方案总结

综合计算成本、存储成本、可复现性和工程复杂度，推荐采用以下方案：

```text
原始行情全量版本化保存
+ 复权因子 PIT 版本化保存
+ 前复权价格按需/增量物化
+ 高维因子结果按版本增量存储
+ 查询层强制使用 trade_date + asof_date 双时间约束
```

该方案具备以下优势：

1. **严格避免未来函数**  
   回测只能看到当时 `asof_date` 可见的数据。

2. **支持结果复现**  
   任意历史训练、回测、推理任务都可通过版本和血缘信息还原。

3. **降低存储成本**  
   不保存每日全量快照，而保存版本差异和必要物化结果。

4. **提升计算效率**  
   高频访问的前复权价格和因子结果可预计算，避免重复实时计算。

5. **支持增量修复**  
   当复权因子、原始行情或因子表达式发生变化时，只需重算受影响范围。

6. **适合多研究员协作**  
   数据口径统一，回测结果可解释、可审计。

## 16. 实施建议

建议按阶段落地：

### 第一阶段：统一数据口径

1. 建立原始行情表；
2. 建立复权因子 PIT 表；
3. 明确前复权计算公式；
4. 统一 `trade_date` 与 `asof_date` 查询规范。

### 第二阶段：建设物化层

1. 对日频前复权价格进行增量物化；
2. 对核心价量因子进行预计算；
3. 建立版本号和计算批次 ID；
4. 建立基础数据质量校验。

### 第三阶段：完善高维因子平台

1. 支持因子表达式版本管理；
2. 支持因子血缘追踪；
3. 支持受影响区间自动识别；
4. 支持回测任务自动绑定数据版本；
5. 支持冷、温、热数据分层。

### 第四阶段：审计与治理

1. 建立数据发布流程；
2. 建立异常告警机制；
3. 建立历史任务复现机制；
4. 建立研究员查询规范；
5. 建立数据修复审批和回滚机制。

---

## 17. 最终结论

对于高维因子系统，不建议采用“实时全量计算”或“永远使用最新前复权”的简单方案。前者计算成本过高，后者会破坏回测的时点一致性并引入未来函数。

每日完整快照虽然最严格，但存储成本和维护成本过高，不适合大规模高维因子长期运行。

最合理的工程方案是采用 **PIT 版本化增量存储架构**：

```text
保存原始数据，版本化复权因子，增量物化前复权价格，版本化存储因子结果。
```

该方案能够在可复现性、性能、存储成本和工程可维护性之间取得平衡，是高维因子生产系统更稳健、更专业的实现路径。
