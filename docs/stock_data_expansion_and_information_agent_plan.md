# 股票数据能力扩展与信息 Agent 建设计划

文档日期：2026-05-29  
适用范围：`tushare-integration` 数据平台、后续量化研究、回测、策略生成与投资信息分析体系。  
文档目标：为管理层提供一份可执行、可分阶段推进、风险可控的建设计划。

---

## 1. 管理层摘要

当前系统已经具备以 Tushare 为核心的数据采集、ClickHouse 落库、DWD/DWS 分层和数据质量校验基础。但如果要进一步支撑更高频的策略研究、更可靠的数据质量治理，以及把重大信息事件纳入策略分析闭环，还需要在三个相对独立的方向上补齐能力：

1. **历史分钟级股票数据与高层数据层建设**
   - 目标是从天级数据扩展到分钟级数据，形成可用于盘中行为研究、事件冲击分析、分钟级回测和高频因子计算的数据基础。

2. **多数据源接入与跨源校验**
   - 目标是从单一 Tushare 数据源扩展为多源体系，通过跨源对账、差异监控和可信源优先级，降低单源错误、接口延迟、字段口径变化对研究和生产策略的影响。

3. **重大信息采集、结构化报告与策略影响分析 Agent**
   - 目标是从“只使用结构化行情/财务数据”扩展为“结构化数据 + 可信信息事件”的分析体系，自动采集新闻、公告、研报、特色市场数据等信息，生成专业报告并持久化，供 AI 和策略评估模块分析其对交易策略的影响。

三个方向可以独立立项、独立交付，也可以共享基础设施，包括数据源注册、任务调度、数据质量控制、元数据管理、血缘追踪和数据库存储规范。建议采用“先核心闭环、再扩展覆盖”的方式推进，避免一开始追求全市场、全频率、全信息源而导致周期失控。

---

## 2. 建设原则

- **先可用，后完备**：优先覆盖 A 股核心股票、核心字段、核心使用场景；稳定后再扩展频率、市场和数据源。
- **源数据忠实落地，业务层统一口径**：ODS 保存原始数据和采集元数据，DWD 做标准化，DWS/ADS 提供面向研究、回测和策略的稳定视图。
- **PIT 与可追溯优先**：所有面向训练、回测、策略生成的数据必须能回答“当时是否可见、来自哪个源、由哪个批次写入、是否被修订”。
- **质量闸口前置**：新增数据必须先进入质量规则、跨源对账和异常记录体系，不能只采集、不验证。
- **数据源可配置、可替换**：不把供应商逻辑写死在业务代码中，通过 source registry 和 connector 插件化降低后续切换成本。
- **AI 只做辅助判断，不直接替代交易决策**：Agent 负责信息采集、摘要、结构化、影响假设和证据整理，最终策略变更必须经过回测、风控和人工审批流程。

## 3. 总体推进路线

| 阶段 | 时间建议 | 重点目标 | 主要产出 |
| --- | ---: | --- | --- |
| 第 0 阶段：调研与方案确认 | 2-3 周 | 确认数据授权、字段范围、存储成本、优先股票池、信息源白名单 | 数据源清单、接口可行性评估、字段映射草案、成本测算 |
| 第 1 阶段：最小可用闭环 | 6-8 周 | 分别打通分钟级数据、多源校验、信息 Agent 的最小闭环 | 1 分钟/5 分钟核心行情表、Tushare 与第二数据源对账、每日信息报告 |
| 第 2 阶段：质量与服务化 | 8-10 周 | 建立 DQC 规则、差异处理流程、报告入库、查询服务 | DWD/DWS 表、质量规则、异常工单表、报告表、API/SQL 服务视图 |
| 第 3 阶段：策略联动 | 10-14 周 | 将分钟级因子、多源置信度、信息事件影响纳入回测和模型分析 | 策略特征集、事件冲击分析、因子有效性评估、策略影响报告 |

> 总体计划周期建议按 7-9 个月管理。时间建议按现有数据平台基础估算，实际周期主要受数据源采购、API 授权、历史数据回补规模、供应商接口稳定性影响。

## 4. 方向一：历史分钟级股票数据与高层数据层建设

### 4.1 为什么需要

当前系统主要围绕天级行情、财务、资金流、指数和基础因子建设。天级数据适合中低频研究，但无法充分刻画以下场景：

- 盘中价格路径、成交节奏、量价背离、尾盘拉升、开盘冲击等分钟级行为。
- 重大新闻、公告、题材发酵后，市场在分钟维度上的反应速度、持续性和衰减路径。
- 更细粒度的止盈止损、滑点估计、成交容量估计和回撤控制。
- 分钟级或日内聚合因子，例如 VWAP 偏离、开盘强度、尾盘强度、日内波动率、成交额集中度、涨跌停附近行为。
- 对天级信号进行盘中执行优化，例如选择更合理的建仓时点、减小冲击成本、过滤异常开盘。

没有分钟级历史数据，策略研究只能看到日线结果，无法解释很多实际交易中的执行差异和短期冲击。

### 4.2 建设后达成的效果

建设完成后，系统应具备以下能力：

- 持续保存 A 股历史分钟级行情，并支持按股票、交易日、时间段快速查询。
- 在 DWD 层提供统一、清洗后的分钟 K 线数据，字段口径不依赖单一数据源命名。
- 在 DWS 层形成分钟级高层特征，包括日内收益、波动率、成交集中度、VWAP、开盘/尾盘强度、异常成交、涨跌停附近行为等。
- 支持分钟级回测、事件冲击分析、盘中执行模拟和天级信号的执行优化。
- 支持将分钟级特征聚合到日级样本中，补充现有 DWS 因子宽表和模型训练集。
- 支持后续与重大信息 Agent 联动，分析某条信息发布后 5 分钟、30 分钟、1 天、3 天的价格与成交影响。

### 4.3 如何实施

#### 4.3.1 数据范围

建议分三步推进：

| 阶段 | 股票范围 | 频率范围 | 历史范围 | 目的 |
| --- | --- | --- | --- | --- |
| MVP | 沪深 300 / 中证 500 / 自选核心股票池 | 1 分钟或 5 分钟 | 最近 1-2 年 | 验证采集、存储、查询、特征计算和成本 |
| 扩展 | 全 A 股普通股票，剔除长期停牌和退市复杂样本 | 1 分钟、5 分钟、15 分钟、30 分钟 | 最近 3-5 年 | 支撑主要研究场景 |
| 完整 | 全 A 股含历史退市样本 | 1 分钟为主，其他周期由 1 分钟聚合 | 尽可能完整 | 支撑严肃回测和生存者偏差控制 |

初期优先保存 1 分钟数据，其他周期通过 DWS 聚合生成，避免重复采集和重复存储。

#### 4.3.2 数据分层

建议沿用现有 ODS/DWD/DWS 思路：

```text
外部分钟数据源
  -> ODS 原始分钟行情
  -> DWD 标准化分钟 K 线
  -> DWS 分钟特征 / 日内聚合特征
  -> ADS 回测与策略输入视图
```

建议新增核心表：

| 层级 | 表名建议 | 主要职责 |
| --- | --- | --- |
| ODS | `ods_stock_minute_bar_raw` | 保存源接口原始字段、原始返回、采集批次、来源、采集时间 |
| DWD | `dwd_stock_minute_bar` | 统一股票代码、交易时间、周期、OHLCV、成交额、复权标记、来源优先级 |
| DWS | `dws_stock_intraday_feature` | 计算分钟级和日内聚合特征 |
| DWS | `dws_stock_event_reaction_minute` | 保存事件发生后不同时间窗口的分钟级反应 |
| ADS | `ads_stock_intraday_factor_panel` | 面向研究、回测和模型训练的稳定访问表 |

#### 4.3.3 核心字段

`dwd_stock_minute_bar` 建议至少包含：

| 字段 | 说明 |
| --- | --- |
| `instrument_id` | 内部统一证券 ID |
| `ts_code` | 源系统股票代码 |
| `trade_date` | 交易日 |
| `trade_datetime` | 分钟归属时间 |
| `bar_interval` | 周期，例如 `1m`、`5m` |
| `open` / `high` / `low` / `close` | 分钟 OHLC |
| `volume` | 成交量 |
| `amount` | 成交额 |
| `vwap` | 分钟 VWAP，可由成交额和成交量计算 |
| `source` | 数据来源 |
| `source_priority` | 当前采用来源优先级 |
| `batch_id` | 采集批次 |
| `ingest_time` | 入库时间 |
| `record_hash` | 行内容哈希，用于识别修订 |
| `is_suspended` | 是否停牌或无交易 |

#### 4.3.4 高层特征建设

第一批 DWS 特征建议聚焦可解释、可回测、计算成本可控的指标：

| 类别 | 特征示例 | 用途 |
| --- | --- | --- |
| 日内收益 | 开盘 5/15/30 分钟收益、尾盘 30 分钟收益 | 捕捉开盘冲击和尾盘资金行为 |
| 日内波动 | 分钟收益标准差、最高最低振幅、上/下行波动 | 衡量短期风险和交易拥挤 |
| 成交结构 | 成交额前 30 分钟占比、尾盘成交占比、成交额集中度 | 判断资金进入时段 |
| VWAP 偏离 | 收盘价相对 VWAP、开盘价相对 VWAP | 优化执行和识别强弱 |
| 流动性 | 分钟成交额中位数、低成交分钟占比 | 判断容量与滑点风险 |
| 涨跌停行为 | 触板时间、封板持续时间、炸板次数 | 服务涨跌停和题材策略 |
| 事件反应 | 信息发布后 5/30/60 分钟收益和成交放大倍数 | 连接信息 Agent 与价格影响分析 |

#### 4.3.5 数据质量规则

新增 DQC 规则应覆盖：

- 唯一性：`instrument_id + trade_datetime + bar_interval + source` 唯一。
- 完整性：交易日内分钟数量符合交易日历和交易时段规则。
- OHLC 规则：`low <= open/close <= high`。
- 非负规则：成交量、成交额非负。
- 价格有效性：有成交时价格必须为正。
- 时间有效性：分钟时间必须落在合法交易时段。
- 跨周期一致性：5 分钟、15 分钟数据应能由 1 分钟数据聚合一致。
- 与天级行情一致性：分钟聚合出的日 OHLCV 与日线行情差异在阈值内。

#### 4.3.6 技术实现要点

- 存储层优先使用 ClickHouse，按 `trade_date` 分区，按 `instrument_id, trade_datetime` 排序。
- 历史回补与每日增量分离，避免历史任务影响日常任务 SLA。
- 对分钟数据采用分批写入和幂等 upsert，避免重复回补造成膨胀。
- 对所有分钟表保留 `source`、`batch_id`、`ingest_time`、`record_hash`，为跨源校验和修订追踪预留能力。
- 优先建设 SQL 可复现的特征计算链路，复杂特征再引入 Python 或 UDF。

### 4.4 实施风险点

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 历史分钟数据量大 | 存储成本、回补时间和查询性能压力明显增加 | 先做核心股票池 MVP；按日期分区；冷热分层；只保存 1 分钟，其他周期聚合 |
| 数据授权限制 | 部分供应商对分钟历史、批量下载、商用使用有限制 | 第 0 阶段完成授权审查；采购前确认 API 频控、历史范围和落库权限 |
| 源数据口径差异 | 不同源对成交量单位、复权、停牌分钟处理可能不同 | DWD 做统一口径；保留源字段；建立跨源差异规则 |
| 回补任务不稳定 | 大批量历史拉取容易中断或超限 | 支持断点续传、任务分片、失败重试和进度表 |
| 与日线数据不一致 | 会影响策略信任度和回测结果 | 建立分钟聚合日线 vs 日线源数据的 DQC 规则 |
| 过早建设复杂高频能力 | 会增加系统复杂度和成本 | 第一阶段只支持分钟级研究和执行优化，不直接承诺 Tick 级或低延迟交易 |

## 5. 方向二：多数据源接入与跨源校验

### 5.1 为什么需要

当前 Tushare 是唯一数据源。单源方案在早期建设效率高，但长期存在以下问题：

- 单一接口异常、延迟、字段变更或权限变化，会直接影响下游研究和生产策略。
- 源数据本身可能存在修订、遗漏、单位变化、异常值，单源无法自证正确。
- 对于关键字段，例如行情、复权因子、财务公告、股本、停复牌、涨跌停、指数成分，错误会直接影响因子计算、回测和实盘信号。
- 当策略结果异常时，如果没有第二来源对照，很难判断是市场变化、模型问题还是数据问题。
- 未来分钟级数据、研报、资讯、特色数据接入会自然形成多源体系，需要提前建立统一框架。

多源接入不是为了简单“多买数据”，而是为了建立数据置信度、稳定性和可审计能力。

### 5.2 建设后达成的效果

建设完成后，系统应具备以下能力：

- 支持 Tushare 之外的至少一个第二数据源，并可继续扩展 Wind、Choice、iFinD、交易所、巨潮资讯、聚源、CSMAR 等数据源。
- 对关键数据集进行跨源对账，包括行情、复权因子、交易日历、证券主数据、财务指标、公告事件、分钟行情等。
- 对差异结果进行分级：可接受差异、需关注差异、阻断级差异。
- 为下游提供“可信主数据视图”，而不是让研究人员自行判断哪个源可用。
- 当某个源异常时，可以降级到备用源或延迟发布，并记录原因。
- 建立供应商质量评估指标，包括覆盖率、延迟、缺失率、异常率、修订频率和差异率。

### 5.3 如何实施

#### 5.3.1 数据源分层

建议将数据源分为三类：

| 级别 | 类型 | 示例 | 使用定位 |
| --- | --- | --- | --- |
| S 级 | 官方或法定披露源 | 交易所、巨潮资讯、上市公司公告、监管机构 | 公告、披露、交易规则、证券状态的权威来源 |
| A 级 | 专业金融数据供应商 | Tushare、Wind、Choice、iFinD、聚源、CSMAR | 行情、财务、因子、研报、宏观、行业等结构化数据 |
| B 级 | 市场特色与投资者常用源 | 开盘啦、财联社、东方财富、同花顺特色榜单等 | 情绪、题材、热度、特色标签和市场叙事补充 |

业务上应优先采用 S/A 级来源作为主校验依据，B 级来源用于补充信息和特征，不作为唯一事实依据。

#### 5.3.2 Connector 框架

新增统一数据源接入框架：

```text
source_registry.yaml
  -> connector 插件
  -> ODS 原始落库
  -> DWD 标准化映射
  -> DQC 单源质量校验
  -> cross_source_validation 跨源校验
  -> trusted view 可信视图
```

`source_registry.yaml` 建议配置：

```yaml
sources:
  tushare:
    type: vendor_api
    priority: 50
    enabled: true
    domains: [eod_price, finance, basic, moneyflow]
  exchange_sse:
    type: official_web_or_api
    priority: 100
    enabled: true
    domains: [announcement, trading_calendar, security_status]
  cninfo:
    type: official_disclosure
    priority: 100
    enabled: true
    domains: [announcement, corporate_action]
```

#### 5.3.3 跨源校验范围

第一批建议覆盖最关键、最容易影响策略结果的数据：

| 数据域 | 校验对象 | 校验方式 | 优先级 |
| --- | --- | --- | --- |
| 交易日历 | 开市日、休市日、特殊交易日 | 集合一致性 | 高 |
| 证券主数据 | 代码、简称、上市日、退市日、交易所、状态 | 主键与字段一致性 | 高 |
| 日线行情 | OHLCV、成交额、涨跌幅 | 精确/容差校验 | 高 |
| 复权因子 | adj factor、分红送转影响 | 变化点校验 | 高 |
| 涨跌停 | 涨跌停价、涨停状态、连板信息 | 字段一致性和规则复算 | 高 |
| 财务公告 | 报告期、公告日、修订时间 | PIT 可见性校验 | 高 |
| 股本与企业行为 | 总股本、流通股、分红、配股、回购 | 事件链一致性 | 中高 |
| 分钟行情 | 分钟 OHLCV、聚合日线 | 容差校验 | 中高 |
| 特色数据 | 题材、热榜、资金行为 | 覆盖率和方向一致性 | 中 |

#### 5.3.4 差异处理机制

建议建立差异表和处理流程：

| 表名建议 | 说明 |
| --- | --- |
| `dq_cross_source_run` | 每次跨源校验运行记录 |
| `dq_cross_source_result` | 汇总级结果，记录数据域、源组合、差异率、严重级别 |
| `dq_cross_source_diff` | 明细差异，记录主键、字段、源 A 值、源 B 值、差异类型 |
| `dq_source_quality_metric` | 按来源沉淀覆盖率、延迟、异常率、差异率等长期指标 |
| `dwd_source_resolution_rule` | 记录不同数据域的主源、备源和冲突裁决规则 |

差异分级建议：

| 级别 | 说明 | 处理方式 |
| --- | --- | --- |
| BLOCKER | 影响交易日、证券主键、价格核心字段、PIT 可见性的重大差异 | 阻断发布或降级发布 |
| WARN | 存在差异但不一定影响核心策略，例如少量成交额容差外 | 发布但告警，进入待处理队列 |
| MONITOR | 仅用于长期观察，例如特色标签覆盖率差异 | 记录趋势，不阻断 |

#### 5.3.5 可信视图

下游不应直接消费多张源表，而应通过可信视图访问：

```text
多个源的 DWD 标准化表
  -> source_resolution_rule
  -> trusted DWD/DWS view
  -> 研究、回测、策略、AI 分析
```

可信视图应明确：

- 每个字段来自哪个源。
- 是否经过跨源校验。
- 如果存在差异，采用哪个裁决规则。
- 是否有降级或异常标记。

### 5.4 实施风险点

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 数据源采购和授权周期长 | 项目启动后可能无法立即接入第二源 | 第 0 阶段并行推进商务与技术 PoC；优先选择已有授权或试用源 |
| 不同源字段口径不同 | 容易出现“看似差异，实为口径不同” | 建立字段字典、单位转换、复权口径说明和容差规则 |
| 跨源校验过于严格 | 可能造成大量误报，影响日常发布 | 初期使用 `warn_only` 建立基线，逐步提升到 `strict` |
| 供应商接口频控 | 大规模对账和回补可能触发限制 | 批量接口优先；缓存原始结果；按数据域和日期分片 |
| 冲突裁决责任不清 | 数据差异无法闭环，影响可信视图使用 | 明确每个数据域的 owner、主源、备源、阻断规则 |
| 成本增加 | 多源采购和存储成本上升 | 先覆盖关键数据域，不做全量重复采购；用质量收益反推采购优先级 |

## 6. 方向三：重大信息采集、结构化报告与策略影响分析 Agent

### 6.1 为什么需要

股价不仅受历史行情和财务数据影响，也会受到重大信息事件驱动，包括政策变化、行业事件、上市公司公告、研报观点、订单中标、并购重组、监管处罚、业绩预告、市场题材、资金偏好和投资者关注度变化。

当前系统如果只依赖结构化行情和财务数据，会存在以下不足：

- 无法及时识别导致价格异动的外部信息原因。
- 无法把新闻、公告、研报、题材和特色数据结构化沉淀，后续无法复盘其对策略的影响。
- 研究人员依赖人工阅读，效率低、标准不统一、可追溯性弱。
- AI 无法系统性学习“某类信息出现后，对特定行业、股票、策略因子是否有稳定影响”。
- 当策略出现异常收益或异常亏损时，缺少信息层证据来解释。

因此需要建设一个可配置的信息 Agent，把可信来源的信息采集、结构化、报告生成、入库和策略影响分析串成闭环。

### 6.2 建设后达成的效果

建设完成后，系统应具备以下能力：

- 按配置采集新闻、公告、研报、政策、行业事件、特色市场数据，例如“开盘啦的明天炒什么”等。
- 数据源必须可配置、可禁用、可分级，并优先选择可信、知名、投资者广泛使用的来源。
- 自动完成去重、实体识别、事件分类、情绪/影响方向判断、证据抽取和重要性评分。
- 每日或盘中生成专业、结构化、可追溯的信息报告。
- 报告末尾给出面向投资参考的综合总结，包括主要矛盾、潜在受益/受损方向、不确定性和需要持续跟踪的信号。
- 所有原始信息、结构化事件、生成报告、引用证据和模型判断必须持久化到数据库。
- 后续 AI 可以基于持久化数据分析这些信息是否对既有交易策略有影响，例如提升收益、增加波动、改变持仓风险、触发风控或改变选股排序。

### 6.3 如何实施

#### 6.3.1 信息类别

信息类别必须配置化，第一批建议覆盖：

| 类别 | 示例 | 主要用途 |
| --- | --- | --- |
| 公司公告 | 业绩预告、重大合同、并购重组、减持、回购、处罚、诉讼、担保 | 识别公司级事件 |
| 财经新闻 | 宏观政策、监管动态、行业事件、上市公司新闻 | 识别市场叙事和外部冲击 |
| 研报观点 | 券商研报、评级变动、盈利预测调整、行业深度 | 识别机构预期变化 |
| 特色数据 | 开盘啦“明天炒什么”、题材热度、连板梯队、资金偏好 | 捕捉短线主题和情绪 |
| 市场异动 | 龙虎榜、涨跌停、成交放量、异动公告 | 解释价格行为 |
| 舆情与关注度 | 搜索热度、互动平台、投资者问答、新闻转载热度 | 识别关注度变化 |

#### 6.3.2 数据源选择原则

数据源进入白名单前必须评估：

- 是否具有明确主体、稳定域名、持续运营记录。
- 是否为官方披露平台、专业金融数据供应商、主流财经媒体或投资者广泛使用的数据工具。
- 是否允许自动化采集、API 调用、内部存储和二次分析。
- 是否提供发布时间、原文链接、唯一 ID、证券代码、行业标签等可追溯字段。
- 是否可以稳定增量更新，避免只能人工浏览。

候选来源分层：

| 级别 | 候选来源 | 主要信息 |
| --- | --- | --- |
| S 级 | 上交所、深交所、北交所、巨潮资讯、证监会、交易商协会等 | 官方公告、监管信息、规则变化 |
| A 级 | Wind、Choice、iFinD、Tushare Pro、聚源、CSMAR | 结构化新闻、公告、研报、财务、市场数据 |
| A 级 | 财联社、中国证券报、证券时报、上海证券报等主流财经证券媒体 | 快讯、深度报道、政策和行业资讯 |
| B 级 | 开盘啦、东方财富、同花顺特色数据、雪球等 | 题材、热度、短线情绪、投资者关注度 |

B 级数据只能作为市场情绪和特色特征输入，不能单独作为事实判断依据；涉及公告、财务、监管处罚、重大合同等事实，应回链到 S 级或 A 级来源确认。

#### 6.3.3 Agent 架构

建议架构如下：

```text
source_config.yaml
  -> crawler / API connector
  -> raw information store
  -> dedup & normalization
  -> entity linking
  -> event classification
  -> credibility scoring
  -> impact hypothesis generation
  -> structured report generation
  -> database persistence
  -> strategy impact analysis
```

模块职责：

| 模块 | 职责 |
| --- | --- |
| Source Registry | 管理数据源、类别、权限、频率、优先级、启停状态 |
| Collector | 通过 API、RSS、授权接口或合规爬虫采集原始信息 |
| Normalizer | 统一标题、正文、发布时间、来源、证券代码、行业、链接、原文哈希 |
| Deduplicator | 基于标题、正文哈希、来源 ID、相似度做去重和聚合 |
| Entity Linker | 将信息关联到股票、指数、行业、概念、人物、机构、地区 |
| Event Classifier | 分类为政策、公告、研报、订单、处罚、并购、业绩、题材等 |
| Scorer | 计算可信度、重要性、时效性、价格影响可能性和不确定性 |
| Report Writer | 生成结构化日报、盘中快报或专题报告 |
| Persistence | 将原文、事件、报告、引用证据、模型输出全部入库 |
| Impact Analyzer | 连接行情、分钟数据、持仓、回测和策略表现，分析信息影响 |

#### 6.3.4 数据库设计

建议新增表：

| 表名建议 | 说明 |
| --- | --- |
| `ods_information_item_raw` | 原始新闻、公告、研报、特色数据，保留原文和采集元数据 |
| `dwd_information_event` | 标准化事件表，包含事件类型、关联标的、发布时间、来源、可信度 |
| `dwd_information_entity_link` | 信息与股票、行业、概念、机构等实体的关联关系 |
| `dws_information_impact_signal` | 事件影响信号，包括方向、强度、时间窗口、置信度 |
| `ads_information_daily_report` | 每日结构化报告和摘要 |
| `ads_information_strategy_impact` | 信息事件与策略收益、回撤、换手、持仓变化的关联分析 |

核心字段建议：

| 字段 | 说明 |
| --- | --- |
| `source` | 信息来源 |
| `source_level` | S/A/B 级 |
| `source_item_id` | 来源侧唯一 ID |
| `publish_time` | 信息发布时间 |
| `crawl_time` | 系统采集时间 |
| `available_time` | 系统可用于分析的时间 |
| `title` | 标题 |
| `content_hash` | 内容哈希 |
| `raw_content` | 原文或授权范围内的正文 |
| `summary` | 摘要 |
| `event_type` | 事件类型 |
| `related_instruments` | 关联股票 |
| `related_industries` | 关联行业 |
| `impact_direction` | 正向、负向、中性、不确定 |
| `impact_horizon` | 影响期限，例如 intraday、1d、3d、20d |
| `importance_score` | 重要性评分 |
| `confidence_score` | 置信度评分 |
| `evidence` | 支撑判断的证据片段或引用 ID |
| `model_version` | 生成报告或分类使用的模型版本 |

#### 6.3.5 报告结构

每日信息报告建议固定结构，保证可读、可比较、可入库：

```text
1. 当日核心结论
2. 市场主线与风险偏好
3. 重大宏观/政策信息
4. 行业与主题机会
5. 公司级重大事件
6. 研报与机构观点变化
7. 特色数据观察
8. 重点股票与行业影响清单
9. 与现有策略的潜在关系
10. 风险、不确定性和待验证事项
11. 投资参考总结
```

报告末尾的总结必须满足：

- 明确区分事实、推断和不确定性。
- 给出影响方向和影响期限，而不是泛泛描述。
- 标明最重要的证据来源和冲突信息。
- 提示哪些结论需要后续价格、成交、资金流或公告验证。
- 不直接给出未经回测和风控验证的买卖指令。

#### 6.3.6 与策略影响分析联动

信息入库后，AI 和研究模块可以做以下分析：

- 事件发生后 `5m/30m/1d/3d/20d` 的收益、成交额、波动率、换手率变化。
- 某类事件对不同行业、不同市值、不同风格股票的平均影响。
- 某类信息是否会增强或削弱现有因子，例如动量、反转、资金流、估值、业绩预期。
- 信息事件是否解释策略异常收益、异常亏损、回撤扩大或持仓集中风险。
- 将信息特征作为模型候选特征，但必须通过样本外测试和严格回测。

### 6.4 实施风险点

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 信息源授权和版权限制 | 原文存储、摘要生成、二次分析可能受限制 | 优先 API/授权源；明确存储范围；必要时只存元数据、摘要和引用 ID |
| 爬虫合规风险 | 非授权采集可能违反网站条款或被封禁 | 使用授权接口、RSS、公开 API；遵守 robots 和访问频率；保留数据源合规记录 |
| 信息噪声高 | Agent 可能输出大量无交易价值信息 | 建立来源分级、去重、重要性评分和白名单类别 |
| LLM 幻觉或过度推断 | 报告结论可能不可靠 | 强制引用证据；区分事实/推断；关键报告引入人工复核；保留模型版本 |
| 实体识别错误 | 新闻可能关联错股票或行业 | 使用证券主数据、简称别名表、人工规则和置信度阈值 |
| 时间可见性错误 | 可能在回测中引入未来信息 | 保存 `publish_time`、`crawl_time`、`available_time`，策略分析只使用当时可见信息 |
| 特色数据稳定性不足 | 开盘啦等特色数据字段、页面或授权可能变化 | 接入层插件化；保留替代源；特色数据只作为 B 级输入 |
| 将报告误用为交易指令 | 可能绕过策略验证和风控 | 报告定位为投资参考和策略分析输入；策略调整必须经过回测与审批 |

## 7. 三个方向的关系与边界

三个方向相互独立，但长期会形成协同：

| 方向 | 独立价值 | 与其他方向的协同 |
| --- | --- | --- |
| 分钟级数据 | 提供更细粒度行情、执行和日内因子能力 | 为信息 Agent 提供事件后价格反应分析；为多源校验增加分钟域 |
| 多源校验 | 提高数据可信度和生产稳定性 | 为分钟数据和信息数据提供来源优先级、差异处理和质量度量 |
| 信息 Agent | 补充非结构化信息和事件解释能力 | 需要分钟数据衡量冲击；需要多源框架保证信息可信 |

边界建议：

- 分钟级数据项目不负责采购所有外部资讯，但需要支持事件反应表。
- 多源校验项目不负责解释市场逻辑，只负责数据可信度和差异治理。
- 信息 Agent 不负责直接下单或替代策略，只负责信息结构化、报告生成和影响分析输入。

---

## 8. 优先级建议

建议按“风险收益比”排序：

1. **优先推进多源校验框架的底座**
   - 原因：分钟级数据和信息 Agent 后续都会依赖数据源注册、血缘、质量规则和可信视图。

2. **并行推进分钟级数据 MVP**
   - 原因：分钟数据对策略研究和事件冲击分析价值高，但数据量和授权风险较大，需要尽早验证成本。

3. **启动信息 Agent 的日报 MVP**
   - 原因：先用少量高可信来源形成结构化日报和入库闭环，再逐步引入研报、特色数据和策略影响分析。

第一阶段不建议做以下事项：

- 不直接建设 Tick 级数据平台。
- 不一次性接入大量低可信网站。
- 不让 Agent 直接产生交易指令。
- 不把所有数据源差异都设为阻断级。
- 不在没有授权确认前批量保存全文研报或受版权保护内容。

## 9. 验收标准

### 9.1 分钟级数据

| 验收项 | 标准 |
| --- | --- |
| 数据覆盖 | MVP 股票池最近 1-2 年分钟数据完整入库 |
| 查询性能 | 单股票一年 1 分钟数据查询可在可接受时间内返回 |
| 数据质量 | OHLC、非负、交易时段、分钟数完整性规则上线 |
| 聚合一致性 | 分钟聚合日线与日线源差异在阈值内 |
| 特征产出 | 至少 10 个第一批日内特征可稳定生成 |

### 9.2 多源校验

| 验收项 | 标准 |
| --- | --- |
| 第二数据源 | 至少接入一个 Tushare 之外的数据源 |
| 校验范围 | 覆盖交易日历、证券主数据、日线行情、复权因子中至少 3 类 |
| 差异记录 | 差异结果可追溯到字段、主键、源值和校验批次 |
| 质量指标 | 能按来源统计覆盖率、差异率、延迟和异常率 |
| 可信视图 | 至少一个核心数据域通过可信视图对下游提供 |

### 9.3 信息 Agent

| 验收项 | 标准 |
| --- | --- |
| 来源配置 | 支持按类别启停数据源，并记录来源级别 |
| 采集闭环 | 至少接入公告、财经新闻、特色数据三类信息 |
| 结构化事件 | 能输出事件类型、关联股票、影响方向、置信度和证据 |
| 报告生成 | 能稳定生成结构化日报并持久化 |
| 策略联动 | 能对事件后价格表现和策略持仓暴露做初步分析 |

## 12. 结论

这三个方向分别解决不同层面的问题：

- 分钟级数据解决“看得更细、测得更准、执行更真实”的问题。
- 多源校验解决“数据是否可信、错误能否发现、生产是否稳定”的问题。
- 信息 Agent 解决“重大信息如何系统化进入研究、报告和策略影响分析”的问题。
